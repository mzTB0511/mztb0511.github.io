    • 一. k8s简单使用

    Kubernetes 是什么

    Kubernetes（简称K8S）是开源的容器集群管理系统，可以实现容器集群的自动化部署、自动扩缩容、维护等功能。它既是一款容器编排工具，也是全新的基于容器技术的分布式架构领先方案。在 Docker技术的基础上，为容器化的应用提供*部署运行、资源调度、服务发现和动态伸缩 *等功能，提高了大规模容器集群管理的便捷性。

    k8s 集群节点（Node）类型

    • 管理节点

      管理节点主要负责K8S集群管理，集群中各节点间的信息交互、任务调度，还负责容器、Pod、NameSpaces、PV等生命周期的管理。

    • 工作节点

      工作节点主要为容器和Pod提供计算资源，Pod及容器全部运行在工作节点上，工作节点通过kubelet服务与管理节点通信以管理容器的生命周期，并与集群其他节点进行通信。

    k8s 能做什么

    • 服务发现和负责均衡

      可以使用 DNS 名称或自己的 IP 地址公开容器服务，如果进入容器的流量很大，可以自动负载均衡并分配网络流量，从而大大提高服务的稳定。

    • 存储编排

      允许自动挂载你选择的存储系统，例如本地存储（磁盘阵列柜、NSF等）、公共云提供商（滴滴、华为、阿里等）等

    • 自动部署和回滚

      你可以使用 Kubernetes 描述已部署容器的所需状态，它可以以受控的速率将实际状态 更改为期望状态。例如，你可以自动化 Kubernetes 来为你的部署创建新容器， 删除现有容器并将它们的所有资源用于新容器。

    • 自动完成资源管理

      允许指定每个容器所需 CPU 和内存（RAM）。 当容器指定了资源限制时，k8s 可以自动来管理容器的资源使用情况。

    • 自我修复

      重新启动失败的容器、替换容器、杀死不响应用户定义的运行状况检查的容器，并且在准备好服务之前不将其通告给客户端

    • 密钥与配置管理

      允许你存储和管理敏感信息，例如密码、OAuth 令牌和 ssh 密钥。 可以在不重建容器镜像的情况下部署和更新密钥和应用程序配置（热更新），也无需在堆栈配置中暴露密钥。

    k8s核心组件

    • 管理节点

      • kube-apiserver

        API 服务器是 k8s 控制面的组件， 是整个k8s 集群对外服务的访问入口，也是通过它暴露出集群的API服务。

        kube-apiserver 设计上考虑了水平伸缩，它可通过部署多个实例进行伸缩。 就是说可以运行 kube-apiserver 的多个实例，并在这些实例之间平衡流量。

      • kube-scheduler

        负责监视新创建的、未指定运行节点（node）的 Pods，选择节点让 Pod 在上面运行。

        调度决策考虑的因素包括单个 Pod 和 Pod 集合的资源需求、硬件/软件/策略约束、亲和性和反亲和性规范、数据位置、工作负载间的干扰和最后时限。

      • kube-controller-manager

        从逻辑上讲，每个控制器都是一个单独的进程， 但是为了降低复杂性，它们都被编译到同一个可执行文件，并在一个进程中运行。

        控制器包括:

        • 节点控制器（Node Controller）: 负责在节点出现故障时进行通知和响应

      • 副本控制器（Replication Controller）: 负责为系统中的每个副本控制器对象维护正确数量的 Pod

        • 端点控制器（Endpoints Controller）: 填充端点(Endpoints)对象(即加入 Service 与 Pod)
        • 服务帐户和令牌控制器（Service Account & Token Controllers）: 为新的命名空间创建默认帐户和 API 访问令牌

      • cloud-controller-manager

        与 kube-controller-manager 类似，cloud-controller-manager 将若干逻辑上独立的 控制回路组合到同一个可执行文件中，供你以同一进程的方式运行。 你可以对其执行水平扩容（运行不止一个副本）以提升性能或者增强容错能力。

        控制器都包含对 云平台驱动 的依赖：

        • 节点控制器（Node Controller）: 用于在节点终止响应后检查云提供商以确定节点是否已被删除
      • 路由控制器（Route Controller）: 用于在底层云基础架构中设置路由
        • 服务控制器（Service Controller）: 用于创建、更新和删除云提供商负载均衡器

      • etcd

        兼具一致性和高可用性的键值数据库，是集群所有集群数据的后台数据库。

    • 工作节点

      • kubelet

        一个在集群中每个节点（node）上运行的代理。 它保证容器（containers）都 运行在 Pod 中。

        kubelet 接收一组通过各类机制提供给它的 PodSpecs，确保这些 PodSpecs 中描述的容器处于运行状态且健康。 kubelet 不会管理不是由 Kubernetes 创建的容器。

      • kube-proxy

        是集群中每个节点上运行的网络代理，实现 k8s 服务（Service） 概念的一部分。

        kube-proxy 维护节点上的网络规则。这些网络规则允许从集群内部或外部的网络会话与 Pod 进行网络通信。

    k8s 常用对象

    • Namespace

      1. 配置格式

        apiVersion: v1
kind: Namespace
metadata:
  name: test

      2. 创建命名空间

        # 创建命名空间
kubectl apply -f namespace.yml 
# 或者
kubectl create namespace test
# 查看所有命名空间
kubectl get namespace
# 删除命名空间
kubectl delete namespace test

      3. 多个命名空间下切换

        # 杭高项目stg 环境
kubectl config use-context saas-stg
# 杭高项目dev 环境
kubectl config use-context saas-dev

      4. 设置命名空间偏好

        # 好处：设置完后访问资源可以省去 -n <namespace> 的麻烦
kubectl config set-context --current --namespace=saas-dev

    • Deployment

      1. 配置说明：

        apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  namespace: test
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 1 # 运行的副本数量，由 副本控制器 控制
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx   #容器名
        image: nginx:1.14.2 # 镜像名
        imagePullPolicy: IfNotPresent  #Allways
        ports:              # 服务域名端口
        - containerPort: 80
      #nodeSelector:   # Node选择器，调度 pod 到指定的Node节点
      #  type: data

      2. 创建Deployment

        # 创建 Deployment, 修改yml文件后，如果有更新，会自动执行
kubectl apply -f deployment.yml

      3. 控制pod副本自动伸缩

        kubectl scale deployment nginx --replicas=2

    • Service

      三种常用类型的Service

      1. ClusterIP

        • 普通Service：通过为Kubernetes的Service分配一个集群内部可访问的固定虚拟IP（Cluster IP），实现集群内的访问。为最常见的方式。

        • Headless Service：该服务不会分配Cluster IP，也不通过kube-proxy做反向代理和负载均衡。而是通过DNS提供稳定的络ID来访问，DNS会将headless service的后端直接解析为podIP列表。主要供StatefulSet使用。

        配置说明：

        # 集群IP
apiVersion: v1
kind: Service
metadata:
  name: nginx
  namespace: test
spec:
  type: ClusterIP
  ports:
  - name:  http
    port:  80
  selector:
    app: nginx

    1. NodePort

      除了使用Cluster IP之外，还通过将service的port映射到集群内每个节点的相同一个端口，实现通过nodeIP:nodePort从集群外访问服务。

      配置说明:

      # NodeIP
apiVersion: v1
kind: Service
metadata:
  name: nginx
  namespace: test
spec:
  type: NodePort
  ports:
  - name:  http
    port:  80
    nodePort: 31100
  selector:
    app: nginx

    2. LoadBalancer

      和nodePort类似，不过除了使用一个Cluster IP和nodePort之外，还会向所使用的公有云申请一个负载均衡器(负载均衡器后端映射到各节点的nodePort)，实现从集群外通过LB访问服务。

      配置说明：

       # NodeIP
 apiVersion: v1
 kind: Service
 metadata:
   name: nginx
   namespace: test
 spec:
   type: LoadBalancer
   ports:
   - name:  http
     port:  80
     nodePort: 31100
   selector:
     app: nginx

    • Statefulsets

      StatefulSet 是用来管理有状态应用的工作负载 API 对象。

      1. 创建存储卷 PVC 

        apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-storage
  namespace: test
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
  storageClassName: ceph-rbd #csi-huawei-lun

      2. 创建Statefulsets

        apiVersion: v1
kind: Service
metadata:
  name: nginx-sset
  namespace: test
  labels:
    app: nginx-sset
spec:
  ports:
  - port: 80
    name: nginx-sset
  clusterIP: None
  selector:
    app: nginx-sset
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: nginx-sset
  namespace: test
spec:
  serviceName: "nginx-sset"
  replicas: 1
  selector:
    matchLabels:
      app: nginx-sset
  template:
    metadata:
      labels:
        app: nginx-sset
    spec:
      containers:
      - name: nginx-sset
        image: nginx:1.14.2
        ports:
        - containerPort: 80
        volumeMounts:  # 容器内部服务路径 
        - name: www
          mountPath: /usr/share/nginx/html
      volumes:         # 容器外部存储介质、目录
        - name:  www
         #mountPath:  主机目录
            #hostPath: /data/xxxx
          persistentVolumeClaim:
            claimName: test-storage

    StatefulSet 中的 Pod 拥有一个唯一的顺序索引和稳定的网络身份标识。

      # kubectl get pods
  nginx-66b6c48dd5-7dgdd   1/1     Running   0          15m
  nginx-sset-0             1/1     Running   0          58m

    • 每个 Pod 都拥有一个基于其顺序索引的稳定的主机名

    • 它创建了一个 Headless Service nginx-sset 用来发布 StatefulSet nginx-sset 中的 Pod 的 IP 地址。

      访问Pods、Service 的访问路径

    • {podName}.{serviceName}.{namespace:default}.svc.cluster.local

    • {serviceName}.{namespace}.svc.cluster.local

      # 访问单个Pod 服务
curl nginx-sset-0.nginx-sset.test.svc.cluster.local

# 通过Service 访问服务
curl nginx-sset.test.svc.cluster.local

    • Configmap

      4 中方式创建**

    • 通过直接在命令行中指定configmap参数创建，即–from-literal

      kubectl create configmap test-config1 --from-literal=key1=v1 --from-listeral=key2=v1

    • 通过指定文件创建，即将一个配置文件创建为一个ConfigMap–from-file=<文件>

      kubectl create configmap test-config2 --from-file=nginx-config.conf

    • 通过指定目录创建，即将一个目录下的所有配置文件创建为一个ConfigMap，–from-file=<目录>

      kubectl create configmap test-config2 --from-file=config

    • 事先写好标准的configmap的yaml文件，然后kubectl create -f 创建

      apiVersion: v1
kind: ConfigMap
metadata:
  name: config-map  #配置文件名称
  namespace: test
data:               #配置数据项可多个。
  nginx-config: |  
    server {
      listen       80;
      server_name  localhost;
      location / {
          root   /usr/share/nginx/html;
          index  index.html index.htm;
      }
      error_page   500 502 503 504  /50x.html;
      location = /50x.html {
          root   /usr/share/nginx/html;
      }
    }
  index.html: |
    <h1>Hellow k8s !!!!!</h1>

    在Deployment中使用

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 1 # 运行的副本数量，由 副本控制器 控制
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx   #容器名
        image: nginx:1.14.2 # 镜像名
        ports:              # 服务域名端口
        - containerPort: 80
        resources:
          limits:
            cpu: "0.2"
            memory: 1Gi
          requests:
            cpu: "0.2"
            memory: 1Gi 
        volumeMounts:
        - name: nginx-config       # nginx config目录
          mountPath: /etc/nginx/conf.d/    
        - name: www-index                    # www 入口页
          mountPath: /usr/share/nginx/html/  
      volumes:
        - name:  nginx-config     
          configMap:
            name: config-map
            items:
              - key: nginx-config    # nginx默认配置
                path: default.conf     
        - name:  www-index
          configMap:
            name: config-map                # www 默认配置
            items:
              - key: index.html
                path: index.html 
      #nodeSelector:   # Node选择器，调度 pod 到指定的Node节点
      #  type: data

    • Ingress

      Ingress 公开了从集群外部到集群内服务的 HTTP 和 HTTPS 路由。 流量路由由 Ingress 资源上定义的规则控制

      配置说明:

      apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-nginx
spec:
  rules:
  - host: web.nginx.supremind.info       # 配置的主机名
    http:
      paths:
      - backend:
          service:
            name: nginx
            port:
              name: http
        path: /nginx
        pathType: Prefix
  - host: "*.sset.supremind.info"    #nginx-sset.supremind.info   # 配置的主机名
    http:
      paths:
      - backend:
          service:
            name: nginx-sset
            port:
              name: http
        path: /sset
        pathType: Prefix

      至此集群中服务可以使用 域名的方式访问。

    二. 开发人员常用命令

    1. 查看集群中各对象状态

      # 查看集群 pods 运行情况
kubectl get pods -o wide -n test

# 查看 pods 配置情况 类似docker inspect 命令
kubectl describe pods podsName -n test

# 查询多种类型对象
kubectl get rc,svc,ingress,deployment,pvc -n test

# 查看某个对象的配置文件
kubectl get statefulsets nginx-sset -o yaml -n test

# 更新Deployment Image 版本信
kubectl set image deployment/nginx nginx=nginx:1.14.1 -n test

# 查看日志
kubectl logs nginx-sset-0 --tail 200 -f
